# 1. Purpose and authority

This document is the operating constitution for the SSS–HHH curriculum project.

It governs:

- instructional design;
- lesson structure;
- student and teacher documents;
- assessment;
- accessibility;
- visual design;
- publishing masters;
- HTML editing and printing;
- PDF and optional DOCX outputs;
- file naming;
- versioning;
- source control;
- quality assurance;
- classroom testing;
- future contributions by humans or AI agents.

When a lesson, template, visual decision, or generated document conflicts with this Bible, this Bible controls unless an approved later version explicitly changes the rule.

The completed SSS content audit controls factual interpretation of the game. The approved SSS Curriculum Blueprint controls the overall classroom architecture. This Bible turns those decisions into reusable production standards.

# 2. Project principles

## 2.1 The game is part of the lesson

The curriculum must not treat gameplay as a reward, decoration, or detached activity.

Each case is an interactive investigation in which students:

- encounter a phenomenon;
- gather evidence;
- compare explanations;
- identify a mechanism;
- diagnose a failure;
- evaluate a response;
- communicate a conclusion.

The accompanying materials deepen those actions rather than duplicating game text.

## 2.2 Scientific reasoning matters more than game performance

Academic assessment may consider:

- relevant evidence;
- accuracy;
- reasoning;
- mechanism;
- vocabulary;
- rejection of alternatives;
- communication;
- completion.

Academic assessment must not consider:

- game score;
- rank;
- speed;
- optional-node completion;
- rapport optimization;
- bonus discoveries;
- gaming skill.

## 2.3 Every required activity is individually completable

Students may discuss, compare evidence, or help one another voluntarily.

No student may be penalized for:

- working alone;
- declining a group role;
- being absent from a partner activity;
- having a partner who does not contribute.

No required lesson may depend on a group presentation, collaborative product, or multi-day project.

## 2.4 One case equals one manageable classroom experience

The standard ceiling is 60 minutes, including:

- device setup;
- directions;
- gameplay;
- written work;
- closure.

The target is approximately 20 minutes of gameplay.

No required homework is assumed.

## 2.5 The fictional setting must strengthen—not blur—the science

Every lesson distinguishes among:

- established science;
- plausible extrapolation;
- debated or context-dependent evidence;
- fictional biology or technology.

Fiction may supply a problem. It must not be presented as empirical evidence about Earth.

# 3. Audience and readability

## 3.1 Intended audience

The standard curriculum version serves:

- middle school learners;
- secondary learners;
- adults and general laypeople;
- classroom teachers unfamiliar with the games;
- future graduate-program reviewers.

The standard version should be accessible to a typical middle-school reader without sounding childish to adults.

## 3.2 Writing standard

Student-facing language should:

- use short, direct sentences;
- define specialized terms near first use;
- place the scientific term before the fictional term when both are needed;
- avoid unnecessary academic jargon;
- use active voice where practical;
- keep directions separate from explanatory prose;
- avoid unexplained abbreviations;
- use consistent names for cases, sources, and tasks.

Teacher-facing language may be more formal but should remain direct and usable.

## 3.3 Direction standard

Directions must be:

- numbered;
- chunked;
- written one action at a time;
- understandable when read aloud by text-to-speech;
- free of instructions that depend only on color or position.

Bad:

> Review the materials below and complete the necessary portions while considering the evidence presented throughout the simulation.

Preferred:

> 1. Open the case.  
> 2. Read the mission question.  
> 3. Record evidence only when it helps explain the plant failure.  
> 4. Complete the diagnosis after the game unlocks it.

# 4. Standard lesson architecture

Every ordinary case lesson follows this sequence.

| Phase | Target time | Required purpose |
|---|---:|---|
| Setup | 0–5 min | Open game and materials |
| Launch | 5–10 min | Introduce phenomenon and investigation question |
| Investigation | 10–30 min | Play and gather targeted evidence |
| Evidence processing | 30–38 min | Organize evidence and reject alternatives |
| Diagnosis and CER | 38–48 min | Explain the most defensible cause |
| Application | 48–55 min | Connect to real science or engineering |
| Exit ticket | 55–60 min | Demonstrate independent transfer |

## 4.1 Protected elements

When time is short, do not remove:

- the investigation question;
- evidence processing;
- CER;
- exit ticket.

The optional application or extension is cut first.

## 4.2 Gameplay fallback

If the game cannot be completed, the student must still be able to demonstrate learning through:

- a case briefing;
- a concise evidence digest;
- screenshots or transcripts;
- the normal reasoning task.

Technical failure cannot become academic failure.

# 5. Student mission sheet standard

## 5.1 Default length

The standard student case packet is one double-sided Letter-size sheet.

A second sheet is allowed only for:

- a required dataset;
- a necessary diagram;
- a one-page reading;
- a culmination requiring more synthesis.

Page count must never grow merely because more text is available.

## 5.2 Required sections

Every ordinary student mission sheet includes:

1. Student identification row on the first page only
2. Mission header
3. Investigation question
4. Vocabulary
5. Initial thinking
6. Case-specific evidence task
7. Diagnosis
8. Rejected alternative
9. CER
10. Real-world application
11. Exit ticket
12. Optional extension


## 5.2A Student identification row

The identification row is the first printable element inside the page margins on page 1 of every Student and Accessible worksheet.

Default fields, in order:

1. Name
2. Date
3. Period

The row appears above the Mission Title Block, institution label, insignia, case title, mission question, and all other content.

For a multi-page worksheet, the row appears only on page 1. Continuation pages do not repeat it. Teacher and Answer Key editions do not use the student identification row.

Additional fields may be added only when institutionally required. They must not displace the three default fields or turn the row into a decorative metadata panel.

Digital fields require programmatic labels, keyboard access, and persistence where the master supports fillable student content.

## 5.2B Exact-match word banks

A constrained fill-in-the-blank task must include a word bank when the intended response depends on recalling an exact technical word or phrase that is not obvious from the sentence structure alone.

The bank uses a one-to-one answer contract:

1. one entry per blank;
2. every expected answer included;
3. no decoys or unrelated vocabulary;
4. no extra entries;
5. repeated entries when one answer is used more than once;
6. phrase answers preserved as phrases.

The bank may be reordered so it does not mirror blank order. It remains immediately adjacent to the activity and on the same page.

The Standard and Accessible editions use the same answer set. The Answer Key repeats the same terms or phrases exactly. Open-ended evidence, diagnosis, CER, interpretation, and design prompts do not receive word banks unless the task has intentionally been converted into a constrained response.

## 5.3 Mission header

The mission header follows the student identification row on the first Student and Accessible page. It includes:

- game title;
- campaign;
- case number;
- case title;
- setting;
- version or compatibility label in small print.

Student Name, Date, and Period are not placed inside or below the Mission Title Block; they remain in the topmost identification row.

## 5.3.1 Science-focus box

Every student mission sheet should identify the lesson's major science focus in plain language near the mission question.

The student-facing box should include:

- the main science concept or phenomenon;
- one concise explanation of what students are investigating;
- a brief science-status line when the case mixes established science with fictional context.

Detailed NGSS codes do not belong in this student box by default. Student materials may instead name the science practice in plain language, such as “Use evidence to explain a pattern and propose a solution.”

The teacher lesson plan must provide the formal NGSS focus separately, including the major SEP, DCI, CCC, relevant performance-expectation connections, and honest limits on the alignment claim.

## 5.4 Initial thinking

Initial thinking is brief and ungraded for correctness.

It may ask students to:

- make a preliminary prediction;
- identify needed evidence;
- identify a likely system;
- distinguish observation from inference.

It should not reveal the diagnosis.

## 5.5 Evidence task

The evidence task is different for each case.

Approved formats include:

- evidence-source matrix;
- timeline;
- cause-and-effect chain;
- competing-hypotheses table;
- systems model;
- sensor/spectrum comparison;
- structure/function diagram;
- design-constraint analysis;
- uncertainty ranking;
- policy evidence map.

A generic clue-transcription log is not an acceptable primary activity.

## 5.6 Diagnosis section

Students must:

- state or select the diagnosis;
- identify the mechanism;
- reject at least one plausible alternative.

## 5.7 CER standard

Every ordinary case includes one compact CER.

Expected standard length:

- Claim: 1–2 sentences
- Evidence: 2–3 specific pieces
- Reasoning: 2–4 sentences

A CER must not be inflated into a long essay.

Evidence should come from more than one source when the case supports that.

## 5.8 Exit ticket

The exit ticket must measure transfer, not recall.

Weak:

> What was the answer to the case?

Stronger:

> A second plant grows sideways in microgravity even though it has enough water and nutrients. What kind of cue could engineers provide, and why would it help?

## 5.9 Optional extension

Extensions:

- must be visibly marked optional;
- must not be required for full credit;
- should fit on the existing page;
- may ask students to evaluate uncertainty, redesign a system, or compare an Earth example.

# 6. Teacher-document standard

Every case eventually receives the following controlled source documents.

## 6.1 Formal lesson plan

Target length: 3–5 pages.

Required components:

- title and overview;
- audience;
- course context;
- duration;
- standards;
- measurable objectives;
- success criteria;
- vocabulary;
- prerequisite knowledge;
- materials;
- technology;
- teacher preparation;
- lesson procedure by time;
- checks for understanding;
- assessment;
- differentiation;
- accessibility;
- misconceptions;
- science/fiction boundaries;
- real-world relevance;
- technical fallback;
- references.

## 6.2 Quick-start guide

Target length: one page.

Required components:

- lesson purpose;
- materials;
- direct game/case access;
- timing;
- correct diagnosis;
- essential science;
- likely sticking point;
- what to collect;
- fallback procedure.

## 6.3 Teacher case analysis

Target length: 1–2 pages.

Required components:

- story problem;
- science phenomenon;
- evidence sources;
- essential evidence;
- optional evidence;
- distractors;
- correct mechanism;
- tempting wrong conclusions;
- instructional opportunities;
- established/extrapolated/debated/fiction labels.

## 6.4 Answer key

The answer key includes:

- accepted evidence;
- model diagnosis;
- model CER;
- reasonable alternative wording;
- common incomplete responses;
- exit-ticket answer;
- annotated student-sheet version where useful.

## 6.5 References

Every important scientific claim in teacher materials is sourced.

Student materials normally use a simple source line rather than formal citations.

# 7. Assessment rules

## 7.1 Required assessment dimensions

The standard quick rubric assesses:

1. Evidence
2. Mechanism
3. Reasoning
4. Communication/completion

The formal rubric may additionally separate:

- source diversity;
- alternative-explanation evaluation;
- vocabulary;
- uncertainty.

## 7.2 No printed numeric points by default

Student pages do not display numeric point values unless a future teacher edition explicitly enables them.

Use:

- success criteria;
- Secure / Developing / Beginning;
- concise descriptive feedback.

## 7.3 Pre/post structure

The SSS curriculum includes:

- a pre-assessment before Campaign 1;
- campaign culminations;
- a final post-assessment/certification after both campaigns.

The pre- and post-assessments measure parallel reasoning skills without repeating the same facts.

## 7.4 Research readiness

Materials should make future evaluation possible by preserving:

- stable task versions;
- consistent rubric categories;
- completion dates;
- optional teacher observation fields;
- version numbers;
- clear distinctions between gameplay and academic scores.

No student research data is collected automatically by the HTML documents.

# 8. NGSS alignment rules

## 8.1 Honest alignment

Do not claim a lesson fully meets a performance expectation merely because the topic appears in the case.

A full alignment claim requires the assessed student product to perform the expectation.

Otherwise use:

- supports components of;
- uses the associated SEP;
- provides context for;
- addresses part of;
- does not by itself fulfill the complete PE.

## 8.2 Three-dimensional alignment

Each lesson identifies:

- Science and Engineering Practice;
- Disciplinary Core Idea or supporting component;
- Crosscutting Concept;
- assessed student product.

## 8.3 Primary standards source

Use official NGSS sources for standards text and identifiers.

Do not copy large copyrighted standards passages into student materials.

# 9. Accessibility and inclusive-design standard

## 9.1 Visual accessibility

Documents must:
- use blank bordered text boxes for extended handwritten responses;
- not place ruled or handwriting lines inside response boxes;

- print clearly in grayscale;
- avoid meaning conveyed only by color;
- maintain readable contrast;
- use at least 11-point print body text unless a tested exception is approved;
- avoid cramped answer areas;
- provide visible labels;
- use repeated layout patterns.

## 9.2 Text-to-speech compatibility

Source documents and HTML outputs must:

- use real text rather than text embedded in images;
- use logical heading order;
- provide labels for controls;
- use alt text for informative images;
- avoid decorative characters that interrupt reading;
- preserve a sensible reading order.

## 9.3 Reduced-writing support

The standard version may include:

- checkboxes;
- short evidence entries;
- sentence stems;
- diagram labels;
- bullet-point evidence.

It must still require genuine reasoning.

## 9.4 No forced group work

No accessibility or differentiation recommendation may quietly convert the lesson into mandatory group work.

# 10. Visual identity

## 10.1 Design direction

The curriculum uses a hybrid style:

- recognizable as an in-universe mission document;
- clean enough for formal classroom and graduate-program use;
- restrained enough to print economically;
- readable without color.

## 10.2 Visual priorities

Priority order:

1. readability;
2. task clarity;
3. answer space;
4. print reliability;
5. in-universe flavor;
6. decoration.

Decoration must never reduce usable writing space or create toner-heavy pages.

## 10.3 Shared elements

Approved recurring elements:

- mission-file header;
- evidence-source icons;
- science-status labels;
- CER block;
- exit-ticket block;
- small case-specific accent;
- footer with compatibility/version information.

## 10.4 Color

Color versions may use a restrained SSS palette.

Grayscale output must remain fully interpretable.

No required instruction may say only:

- choose the green item;
- follow the red line;
- use the blue box.

Use labels, patterns, icons, or line styles as well.

## 10.5 Game artwork

Game artwork and screenshots may be used selectively when they:

- orient the student;
- show evidence;
- support a diagram;
- strengthen identity.

They must not become decorative clutter.

# 11. Canonical source and publishing strategy

## 11.1 Three-layer model

The project uses three distinct layers.

### Layer 1 — Controlled content source

Preferred formats:

- Markdown for prose and structured curriculum content;
- JSON or YAML only where machine-readable structure adds value;
- shared CSS and JavaScript for publishing behavior.

This layer is optimized for:

- Git history;
- agent editing;
- consistency;
- review;
- reuse.

### Layer 2 — HTML publishing master

The HTML version is the canonical visual-layout and printing master.

It controls:

- page size;
- typography;
- spacing;
- margins;
- page breaks;
- teacher/student views;
- answer-key visibility;
- print behavior;
- fillable controls;
- PDF production.

### Layer 3 — Published outputs

Published outputs include:

- printable PDF;
- fillable PDF where practical;
- optional DOCX;
- optional Google Docs copy;
- downloadable HTML.

PDF is the official fixed release.

DOCX and Google Docs are convenience formats, not the layout authority.

## 11.2 Why Markdown is retained

Markdown is not handed to students.

It remains useful as:

- editable curriculum source;
- version-controlled text;
- agent-readable content;
- a reliable input to publishing tools.

## 11.3 Why HTML is the publishing master

HTML/CSS allows:

- one shared design system;
- global style updates;
- visible Letter-size pages;
- adjustable margins;
- teacher and student modes;
- interactive editing;
- browser printing;
- accessible structure;
- consistent PDF generation.

## 11.4 DOCX rule

DOCX may be generated when:

- a professor or school requires Word;
- a teacher requests a familiar editable copy;
- manual editing is expected.

DOCX export must be labeled as an editable convenience version.

It may not replace the controlled source or HTML publishing master.

# 12. HTML publishing-master requirements

## 12.1 Page model

Default page:

- US Letter;
- 8.5 × 11 inches;
- portrait;
- white page surface;
- visible page boundary on screen;
- no fixed page shadow in print.

Landscape may be used only when a lesson genuinely needs it.

## 12.2 Default margins

Default content margins:

- top: 0.50 in;
- bottom: 0.50 in;
- left: 0.50 in;
- right: 0.50 in.

Teacher packets may use 0.65–0.75 in where readability requires it.

No required content may depend on a printer’s non-printable edge.

## 12.3 Margin controls

Editable HTML masters must allow margin adjustment within safe ranges.

Recommended range:

- 0.25–1.00 in;
- increments of 0.05 or 0.10 in.

Controls should include:

- top;
- bottom;
- left;
- right;
- reset to default.

## 12.4 View toggles

The publishing master should support:

- show/hide page boundaries;
- show/hide margin guides;
- show/hide page-break guides;
- grayscale preview;
- teacher/student view;
- show/hide answer key;
- edit/preview mode;
- print-preview mode.

## 12.5 Editing

Basic edit mode should allow:

- editing nearly all meaningful instructional and descriptive text;
- changing headings, prompts, timing, standards text, objectives, evidence labels, CER prompts, exit tickets, answer-key explanations, rubric descriptors, source lines, headers, footers, and teacher notes;
- increasing or reducing answer space;
- hiding optional sections;
- adding teacher notes;
- undoing changes within the current session where feasible.

All instructional and descriptive text should be editable in teacher edit mode.

The following remain locked by default:

- page-size mechanics;
- margin and page-break guides;
- print and view controls;
- layout containers;
- structural IDs and classes;
- decorative borders and icons;
- system-generated metadata.

Protected text such as case identifiers, standards codes, compatibility labels, scientific disclaimers, and source citations may be unlocked deliberately by the teacher rather than being permanently immutable.

## 12.6 Saving

The HTML master should offer:

- download current HTML;
- export/print to PDF;
- reset to source version.

Browser-local autosave may be included, but it must be transparent and optional.

No student data should be silently uploaded.

## 12.7 PDF export

Primary PDF workflow:

1. Open publishing master.
2. Choose view.
3. Confirm page size and margins.
4. Choose Print / Save PDF.
5. Use Letter paper.
6. Use 100% scale.
7. Disable browser headers/footers.
8. Enable background graphics only when the design requires them.

A future scripted PDF export may supplement but not obscure the browser workflow.

## 12.8 DOCX export

A DOCX export button may be added, but it must warn:

- layout may differ;
- page breaks may move;
- fonts may substitute;
- HTML/PDF remains the official layout.

The project must not delay classroom materials merely to create perfect DOCX conversion.

## 12.9 Print CSS

Every publishing master must use dedicated print styles that:

- remove toolbars and edit controls;
- force Letter dimensions;
- preserve intended page breaks;
- prevent major blocks from splitting where practical;
- remove screen-only guides;
- preserve grayscale readability;
- avoid blank trailing pages;
- keep headers and footers within the page box.

## 12.10 Page overflow

The editor must visibly warn when content exceeds a page.

Overflow must be resolved by:

1. removing unnecessary words;
2. tightening nonessential spacing;
3. moving an optional section;
4. adding a deliberate page;
5. reducing type only as a last resort.

Never solve overflow by shrinking everything until it is difficult to read.

# 16. Versioning

**Approved student-identification clarification:** Name, Date, and Period are the topmost first-page elements in Student and Accessible worksheets and are not repeated on continuation pages.

**v1.1 design clarification:** Extended-response boxes are blank and unruled. Compact sentence-completion blanks may still use a single underline when appropriate.

## 16.1 Version meanings

- `v0.1`, `v0.2`, etc. — working drafts;
- `v1.0` — first approved stable release;
- `v1.1` — backward-compatible correction or improvement;
- `v2.0` — substantial redesign or compatibility break.

## 16.2 Source history

Do not overwrite significant draft milestones before stable release.

Retain:

- audit versions;
- blueprint versions;
- major prototype versions;
- correction specifications.

Routine lesson edits after stable release should be handled through Git history rather than endless duplicate files unless a published version changes.

## 16.3 Status labels

Documents should clearly state:

- Draft;
- Review;
- Approved;
- Deprecated;
- Archived.

- [ ] Investigation question is clear
- [ ] Worksheet does not spoil diagnosis
- [ ] Evidence task requires reasoning

## Accuracy

- [ ] Correct mechanism
- [ ] Claims sourced
- [ ] Fiction clearly separated
- [ ] Uncertainty preserved
- [ ] No serious misconception introduced
- [ ] Game text and curriculum agree

## Accessibility

- [ ] Chunked directions
- [ ] Logical reading order
- [ ] Text-to-speech compatible
- [ ] Grayscale readable
- [ ] No color-only meaning
- [ ] Adequate answer space
- [ ] Technical fallback available

- [ ] Letter-size pages
- [ ] Margins within safe range
- [ ] No accidental overflow
- [ ] Page breaks intentional
- [ ] Student and teacher views correct
- [ ] Answer key hidden in student output
- [ ] PDF prints at 100%
- [ ] Version metadata present

# 19. Agent and contributor rules

AI agents or human contributors must:

- read the current Audit, Blueprint, and Bible before making curriculum changes;
- change only the requested scope;
- preserve established naming and structure;
- avoid inventing science;
- cite important claims;
- distinguish source content from generated output;
- not convert optional activities into requirements;
- not add mandatory group work;
- not use game score as academic assessment;
- not expand lesson duration without owner approval;
- not redesign the publishing system within a case-level task.

When uncertainty affects story canon, learning objectives, evidence availability, or scientific accuracy, ask the owner.

Minor formatting choices may use best judgment.
